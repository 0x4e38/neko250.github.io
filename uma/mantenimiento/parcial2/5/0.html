Pruebas de sistema

  • Características clave:
    • Completa (el sistema completo, la especificación completa)
    • Basada en la especificación de comportamiento observable
      • Verificación contra una especificación de requisitos
    • Independiente del diseño y de la implementación
      • Evita repetir en el diseño de las pruebas de sistema errores en el diseño del software
  • Pruebas muy heterogéneas, lo que incluimos en las pruebas es muy dependiente del tipo de aplicación
  • Las pruebas de sistema pueden ser muy costosas y requerir mucho tiempo
    • Puede incluir la construcción de componentes físicos y software para proporcionar el entorno de pruebas necesario (que puede incluso superar el coste del sistema software desarrollado)

Verificación y validación independientes

  • Una estrategia para maximizar la independencia:
    • Pruebas de sistema (y aceptación) realizadas por una organización diferente
      • Aislada organizacionalmente de los desarrolladores (libre de presión)
      • A veces externalizada a otra empresa
        • Especialmente para sitemas críticos
        • Externalizada para tener un juicio independiente, no para ahorrar dinero
        • Puede que solo se externalice parte del análisis, o que se realice uno adicional sin reemplazar el interno
    • En caso de desarrollarlas la misma organización
      • Independencia perfecta no es alcanzable
      • Pero podemos mejorarla desarrollando la especificación de las pruebas cuanto antes
        • Como parte del análisis de requisitos (agile test first o modelo en V)
        • Podemos explotar el que sea la misma organización: "design for test"

Pruebas de sistemas incrementales

  • Las pruebas de sistema suelen utilizarse para medir el progreso del desarrollo
    • Las pruebas de sistema cubren todas las funcionalidades y escenarios de uso
    • Conforme el proyecto progresa el sistema pasa más y más pruebas de sistema
  • Ej: con un plan de desarrollo basado en threads, hay disponibles más funcionalidades conforme se avanza en el desarrollo

Propiedades globales

  • Algunas propiedades de los sistemas osn inherentemente globales
    • rendimiento, latencia, fiabilidad, seguridad...
    • las pruebas tempranas e incrementales son todavía necesarias, pero solo proporcionan estimaciones
  • Objetivo primordial de las pruebas de sistema
    • La única oportunidad para verificar propiedades globales contra las especificaciones del sistema
    • En particular para encontrar efectos imprevistos
      • por ejemplo, un cuello de botella imprevisto en el rendimiento

Propiedades dependientes del contexto

  • Algunas propiedades (globales) dependen del contexto y del uso
    • Ej: el rendimiento puede depender del entorno y de la configuración
    • Ej: la privacidad depende del sistema y de cómo este es utilizado
      • los sistemas de informes médicos deben ser protegidos de usos no autorizados, las autorizaciones deben darse cuando sean necesarias
  • Cuando una propiedad (rendimiento, tiempo de respuesta...) depende de su uso (peticiones por seg., tamaño de BDs...)
    • debemos variar los parámetros dentro de los rangos aceptables, cerca de los bordes, y más allá
  • Objetivo: Modelo de cómo la propiedad varía con el parámetro
    • cómo de sensible es la propiedad con respecto a dicho parámetro
    • cuál es el límite
    • qué podemos esperar cuando se supera este valor

Tipos de pruebas de sistema

  • Pruebas de capacidad
  • Pruebas de estrés
  • Pruebas de usabilidad
  • Pruebas de seguridad
  • Pruebas de rendimiento
  • Pruebas de fiabilidad
  • Pruebas de conformidad
  • Pruebas de disponibilidad/reparabilidad
  • Pruebas de documentación
  • Pruebas de configuración

Pruebas de capacidad

  • Cuándo: los sistemas que deben tratar grandes volúmenes de datos deben tener probados sus límites, y cómo reaccionan ante fallos cuando se sobrepasa esa capacidad
  • Qué/Cómo: construiremos un arnés que genere una gran cantidad de datos, bien simulados o con datos existentes
  • Por qué: debemos asegurarnos de que el sistema es capaz de manejar el volumen de datos para el que fue pensado
  • A favor: proporciona confianza en que el sistema es capaz de manejar volúmenes elevados de datos
  • En contra: puede ser complicado crear datos representativos, puede llevar mucho tiempo hacer las pruebas

Pruebas de estrés

  • Cuándo: sistemas que deben responder en tiempo real (sistemas de control) o sistemas empotrados (sistemas antibloqueo de frenado)
  • Qué/Cómo: tráfico de mensajes muy grande y condiciones extremas. Construimos una estructura en que se integren componentes mecánicos, electrónicos y software
  • Por qué: es necesario tener la certeza de que funcionan en esas condiciones para que se pueda confiar en ellos
  • A favor: inevitable para este tipo de sistemas
  • En contra: es posible que no se pueda reproducir las condiciones y es fácil omitir algún componente

Pruebas de usabilidad

  • Cuándo: sistemas con una interfaz de usuario significativa y donde es importante evitar errores de uso (cuadro de mandos de un avión, sistema de entretenimiento...)
  • Qué/Cómo: podríamos construir un simulador, o simplemente hacer que lo prueben muchos usuarios en un entorno controlado. Es necesario estructurar las pruebas con objetivos claros y tener formas adecuadas de recopilar y analizar datos
  • Por qué: puede haber cuestiones de seguridad o simplemente hacer de la usabilidad un valor del producto
  • A favor: en contextos controlados puede producir buen feedback, a menudo respaldado por alguna teoría (estimaciones de carga cognitiva, etc.)
  • En contra: ciertos requisitos de usabilidad son difíciles de expresar y probar, es posible que se prueben extensivamente y después no se sepa qué hacer con los datos

Pruebas de seguridad

  • Cuándo: la mayoría de sistemas abiertos, para evitar usuarios malintencionados
  • Qué/Cómo: un amplio abanico de posibilidades: tablas de errores típicos a comprobar y revisar, preparar equipos que intenten romper el sistema...
  • Por qué: es esencial para ciertos sistemas
  • A favor: el mayor esfuerzo ha de hacerse en el diseño del sistema y en el uso de componentes seguros
  • En contra: si nos limitamos a listas de errores habituales estamos desprotegidos ante nuevos tipos de ataques

Pruebas de rendimiento

  • Cuándo: muchos sistemas están obligados a satisfacer determinados niveles de rendimiento de acuerdo con sus contratos a nivel de servicio
  • Qué/Cómo: dos enfoques:
    • modelado/simulación
    • ejecución directa en un entorno simulado (o en el entorno real)
  • Por qué: muchas compañías cobran por ofrecer un determinado nivel de servicio (ej: el sistema de pagos de VISA garantiza autorizaciones de servicio en menos de 5s)
  • A favor: muestra evidencias del rendimiento del sistema, el modelado puede indentificar cuellos de botella y otros problemas
  • En contra: no es fácil encontrar pruebas realmente significativas

Pruebas de fiabilidad

  • Cuándo: necesitamos probar que ciertos sistemas fallarán con muy poca frecuencia (ej: sistema de control de una central nuclear no fallará más de una vez por 10000 horas de funcionamiento)
  • Qué/Cómo: creamos una prueba relevante y recopilamos información suficiente para apoyar la afirmación estadística
  • Por qué: sistemas regulados o porque queremos hacer de la fiabilidad un valor añadido
  • A favor: si los datos de prueba son representativos es posible hacer predicciones precisas
  • En contra: es necesario recopilar enormes cantidades de datos para sistemas de alta fiabilidad

Pruebas de conformidad

  • Cuándo: en mercados regulados es necesario demostrar que el sistema es conforme con las especificaciones (ej: un compilador de C debe ser conforme con ANSI C)
  • Qué/Cómo: es habitual que haya conjuntos de pruebas estandarizadas que consituyen buenos niveles de cobertura del comportamiento del sistema
  • Por qué: podemos identificar áreas problemáticas y proporcionar pruebas para comprobar dichas condiciones
  • A favor: el coste de las pruebas es compartido
  • En contra: normalmente el desarrollo se orienta a superar las pruebas y se cuidan menos otros aspectos

Pruebas de disponibilidad/reparabilidad

  • Cuándo: estamos interesados en conocer la frecuencia con que ocurren los fallos y el tiempo que tarda el sistema en recuperarse (ej: proveedor de servicio)
  • Qué/Cómo: similar a los tests de fiabilidad, aunque en este caso se está más interesado en conocer cuánto tarda en recuperarse
  • Por qué: no queremos largas interrupciones en servicios críticos
  • A favor: similar a los de fiabilidad
  • En contra: similar a los de fiabilidad (puede ser más fácil recuperarse de errores conocidos)

Pruebas de documentación

  • Cuándo: la documentación de todos los sistemas con documentación debería ser probada
  • Qué/Cómo: la documentación se corresponde con el sistema. Bastaría con que alguien siguiera la documentación completa
  • Por qué: los usuarios pueden confundirse si la documentación no se corresponde con el sisteam
  • A favor: garantiza la consistencia
  • En contra: difícil comprobar la consistencia de la narrativa y no solo de los ejemplos

Pruebas de configuración

  • Cuándo: a lo largo de la vida del sistema, cuando la configuración software o hardware son modificadas
  • Qué/Cómo: típicamente establecemos relaciones o restricciones entre los diferentes componentes (ej: bibliotecas y sistemas, hardware y drivers, etc.)
  • Por qué: cambios en la configuración provocarán fallos si no son corregidos
  • A favor: trata un tipo de errores cada vez más frecuente. Es posible automatizar este tipo de comprobaciones
  • En contra: depende de las dependencias que se establezcan

Pruebas de aceptación

  • Propósito: guiar en la decisión de si el producto puede ser distribuido
  • Medidas de calidad, no se buscan errores
  • Los criterios de aceptación son estadísticos
    • fiabilidad
    • disponibilidad
    • tiempo medio entre fallos
    • ...
  • Requiere muestras estadísticas válidas de registros de funcionamiento normal

Muestreo estadístico

  • Necesita un modelo preciso de uso: qué se está midiendo y para qué población
    • una versión previa del sistema
    • un modelo del entorno de actividad (ej: para un controlador empotrado)
    • pruebas de sensibilidad pueden revelar qué parámetros son más importantes
  • Necesita una definición clara de qué se está midiendo
    • ratio de fallo ? por sesión, por hora, por operación ?
    • un sistema que se ejecuta 1 vez por segundo, con 99.9999% de fiabilidad (un ratio de fallos de 1 entre un millón) falla 31 veces en un año !
  • Son necesarias cantidades enormes de muestras aleatorias
    • en especial para niveles elevados de fiabilidad

Medidas basadas en el proceso

  • Menos riguroso que las pruebas estadísticas
    • basadas en la similitud con proyectos anteriores
  • Proceso de pruebas de sistema
    • historia esperada de errores encontrados y resueltos
  • Pruebas alfa y beta
    • pruebas alfa: usuarios reales en entornos controlados
    • pruebas beta: usuarios reales en entornos reales no controlados
    • muestrea estadísticamente usuarios, no usos
    • historia esperada de informes de errores

Usabilidad

  • Qué es un producto usable
    • se aprende rápido
    • permite que los usuarios lo utilicen eficientemente
    • es agradable de usar
  • Criterios
    • tiempo y número de operaciones para realizar una tarea
    • frecuencia de errores de usuario
      • los errores del usuario son culpa del producto !
    • y sobre todo, satisfacción subjetiva
  • La usabilidad es algo que depende de los usuarios reales (validación, no verificación)
  • Es necesario identificar criterios para poder establecer la usabilidad a lo largo del desarrollo del proyecto
    • prototipos incrementales, listas de objetivos...

Variedades de pruebas de usabilidad

  • Pruebas exploratorias
    • modelos mentales de los usuarios
    • realizado previamente para guiar el diseño de interfaces
  • Pruebas de comparación
    • evaluar opciones (opciones de diseño de interfaces)
    • observar (y medir) las interacciones con patrones de interacción alternativos
  • Pruebas de validación de usabilidad
    • estimar la usabilidad global (cualitativa y cuantitativamente)
    • medidas consideradas: ratio de errores, tiempo para completar tareas...

Protocolo de pruebas de usabilidad típico

  • Seleccionar muestra representativa de grupos de usuarios
    • Típicamente 3-5 usuarios de cada uno de 1-4 grupos
    • La pertenencia a cada grupo es verificada con cuestionarios
  • Pedir a los usuarios que realicen una secuencia representativa de tareas
  • Observar, sin interferencia (sin ayuda)
  • Medir (clicks de ratón, movimientos de ojos, tiempo...)
  • Acabar con un cuestrionario (opinión...)

Pruebas de regresión

  • "Ayer funcionaba, hoy no"
    • "Estaba arreglando X y accidentalmente romí Y"
    • "Ese error estaba corregido, ahora ha vuelto"
  • Las pruebas han de ser ejecutadas de nuevo tras cada cambio
    • Al añadir nuevas funcionalidades
    • Al cambiar, adaptar el software a nuevas condiciones
    • Arreglando otros errores
  • Las pruebas de regresión pueden suponer una parte importante de los costes de mantenimiento del software
    • Algunas veces mucho más que hacer el cambio

Problemas de las pruebas de regresión

  • Mantenimiento del suite de pruebas
    • Si cambio una característica, cuántos casos de prueba han de ser revisados ?
    • Qué casos de prueba deberían ser eliminados o reemplazados ?
    • Qué nuevos casos de prueba deberían añadirse ?
  • Coste de re-ejecutar las pruebas
    • A menudo proporcional al tamaño del código, no del cambio
    • Muy costoso si requiere un esfuerzo manual
      • Incluso para pruebas automáticas si el tiempo de ejecución de las pruebas es superior a un par de horas

Mantenimiento de los casos de prueba

  • Es inevitable tener que mantener las pruebas
    • Si ha cambiado la característica X los casos de prueba para X necesitarán ser actualizados
  • Parte del mantenimiento puede evitarse
    • Ej: cambios triviales a las GUIs o formatos de ficheros no deberían invalidar grandes cantidades de pruebas
  • Los suites de pruebas deberían ser modulares
    • Evita dependencias innecesarias
    • El generar casos de pruebas específicos para cada especificación de prueba puede ayudar

Selección de casos de prueba de regresión

  • Debemos re-ejecutar el conjunto de pruebas de regresión completo ? Y si la respuesta es sí, en qué orden ?
    • Si el coste es bajo (tiempo y recursos) adelante
  • El orden importa cuando
    • Es caro ejecutar los casos de prueba
      • requieren equipamiento especial
      • largos tiempos de ejecución
      • no pueden ser automatizadas
  • Es importante dar prioridades cuando
    • Un gran conjunto de pruebas tarda demasiado en ser ejecutado

Selección de pruebas de regresión basada en código

  • Observación: un caso de prueba no puede encontrar un error en código que no ejecuta
    • En un gran sistema, muchas partes del código no son tocadas por muchos casos de prueba
  • Y por tanto: ejecutemos solo casos de prueba que ejecutan código cambiado o nuevo

Selección de casos de prueba basados en flujos de control y datos

  • Misma idea que para la selección basada en código
    • Re-ejecuta casos de prueba solo si incluyen elementos cambiados
    • Los elementos cambiados pueden ser nodos y arcos en los flujos de control, o pares DU en los flujos de datos
  • Para automatizar la selección
    • Hay herramientas que registran los elementos afectados por cada caso de prueba
      • Almacenados en bases de datos de casos de prueba de regresión
    • Hay herramientas que detectan cambios en los programas

Selección de casos de prueba de regresión basados en especificación

  • Seleccionamos casos de prueba que prueban funcionalidades nuevas o alteradas
  • Diferencia. no hay garantía de independencia
    • Un caso que no es "para" una característica cambiada o añadida puede encontrar un error en esa característica
  • Enfoque típico
    • Ejecuta todos los casos de prueba, pero empieza por aquellos que están relacionados con las características cambiadas o añadidas
