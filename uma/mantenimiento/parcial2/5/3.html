Pruebas de aplicaciones web

Particularidades de software para la web

  • Interfaz de usuario con HTML (php, cgi...)
  • Disponible para todo el mundo
  • Aceptan peticiones a través de HTTP y devuelven respuestas
  • HTTP no mantiene estado
    • cada petición/respuesta es independiente
  • Compuesto de componentes software independientes, relativamente pequeños y débilmente acoplados
    • toda la comunicación es mediante envío de mensajes
    • los mensajes siempre van a través de clientes
    • la única memoria compartida viene dada por el objeto sesión (muy restringido)
  • Inherentemente concurrente y distribuido
  • Numerosas nuevas tecnologías, a menudo combinadas

Problema general

  • Las aplicaciones web son heterogeneas, dinámicas y deben satisfacer atributos de calidad muy exigentes
  • La web está plagada de sitios y aplicaciones web de baja calidad
  • Las aplicaciones web necesitan ser construidas mejor y más probadas

Aplicaciones en niveles: separación de aspectos

Particularidades de las pruebas de software web

  • Los grafos tradicionales no son aplicables
    • Grafo de flujo de control
    • Grafo de llamadas
  • Comportamiento basado en estados es difícil de modelar y describir
  • Todas las entradas van a través de la GUI en HTML
    • Baja controlabilidad
  • Complicado conseguir acceso al estado del servidor (memoria, ficheros, bases de datos...)
    • Baja observabilidad
  • Complicado identificar asertos efectivos

Problemas específicos

  • Las aplicaciones web son distribuidas y extremadamente desacopladas
    • Internet y hardware/software muy diverso
  • Los formularios HTML son creados dinámicamente
    • Las GUIs se crean bajo demanda y pueden cambiar dependiendo del usuario y del momento
  • Los usuarios pueden cambiar el flujo de control arbitrariamente
    • Botón de retroceso, botón de avance, reescritura de URL, refresco...
  • Integración dinámica de nuevos componentes software
    • Nuevos componentes pueden ser añadidos en tiempo de ejecución

Hipertexto estático

  • No probamos un programa
    • Tan sólo que los links sonválidos
  • También se comprueba
    • Pruebas de carga
    • Evaluación de rendimiento
    • Cuestiones de control de acceso
  • El modelo utilizado es normalmente un grafo
    • Los nodos son las páginas web y los arcos los enlaces HTML

Hipertexto dinámico

  • La interfaz de usuario está en el cliente
  • Parte del software puede estar también en el cliente (JS)
  • La mayor parte del software está en el servidor
  • Las pruebas del cliente no acceden al servidor
  • Las pruebas del servidor pueden usar los fuentes y el estado del servidor

Pruebas del cliente (caja negra)

  • La GUI y el software están en máquinas diferentes
  • Las entradas del software web se definen por formularios HTML
    • Cajas de texto, botones, listas, enlaces, etc.
  • Técnicas para generar valores
    • A mano por el probador
    • Generadas aleatoriamente
    • Datos de sesiones del usuario recopilada de usuarios anteriores
  • Selección de valores
    • Valores que violan restricciones sobre las entradas, como definidas por la información del cliente

Tipos de validación de entrada del cliente

  • La validación de las entradas en el lado del cliente se llevan a cabo en controles de formularios HTML, sus atributos y scripts en el lado del cliente
  • Restricciones HTML
    • Longitud: máximo de caracteres de entrada
    • Valores: valores prefijados
    • Modo de transferencia: GET o POST
    • Elemento de campo: campos prefijados
    • URL destino: enlaces con valores
  • Restricciones de scripts
    • Tipos de datos: comprobación de enteros
    • Formato de datos: formato del código postal
    • Valor de datos: rango de edades
    • Entre -valores: número de créditos + fecha de expediente
    • Caracteres no válidos: <, ../, &...

Comprobación de las salidas

  • Tipos de respuestas de un servidor
    • (V) Respuestas válidas: entradas no válidas no son tratadas adecuadamente
    • (F) Fallos y errores: entradas no válidas que causan un comportamiento del servidor (típicamente capturadas por el servidor web cuando la aplicación falla para manejar el error)
    • (E) Exposición: las entradas no válidas no son reconocidas por el servidor y el comportamiento del software anormal es expuesto a los usuarios
  • Tanto (F) como (E) son respuestas no válidas

Cómo comprobamos

  • Empezamos tratando el sitio web como una caja negra
    • Construimos un modelo del sitio
  • El texto de las páginas web lo tratamos como documentación
    • Corrección de información de contacto (teléfonos, direcciones...)
    • Fechas y notificaciones de copyright
    • Texto del título, texto del marcapáginas, favoritos del navegador
    • Texto alternativo
    • Cuestiones de los gestores de esquemas
  • Hipertexto:
    • Cada enlace debería ser comprobado (visibles y llevan al sitio esperado)
    • mailto: operativos
    • Páginas huérfanas (páginas no accesibles)
  • Gráficos
    • Los gráficos se cargan y muestran adecuadamente
    • Falta algún gráfico o no tiene un nombre correcto
    • Si la página combina texto y gráficos
      • Qué pasa cuando se reduce el tamaño ? Y si se aumenta ?
    • Carga la página lo suficientemente rápido ?
      • Demasiados gráficos ?
      • Tipo de conexión (módem y LAN de alta velocidad)
  • Formularios
    • Posicionamiento y tamaño de los campos de texto
    • Aceptan datos correctos ? Rechazan datos incorrectos ?
    • Son los campos opcionales realmente opcionales
    • Los ataques de overflow de buffers se hacen típicamente a través de los formularios

Rendimiento y seguridad del servidor

  • Sitios populares pueden recibir muchos accesos
    • YouTube tiene 100 millones de descargas de vídeos al día
  • Cada acceso supone descargar datos del servidor al navegador
    • Es necesario simular millones de conexiones y descargas para probar el rendimiento y la carga del sistema
  • Cuestiones de seguridad
    • Ataques de DoS
    • Ataques de Buffer Overflow

Pruebas de configuración y compatibilidad

  • Plataformas hardware: Mac, PC, SmartPhones, Tablets...
  • Navegadores y versiones: Firefox, IE, Safari, Chrome, Opera...
  • Plug-ins de navegadores: para audio, vídeo...
  • Opciones de navegadores: seguridad, plug-ins, pop-ups...
  • Resolución de pantalla: 640x480, 800x600, 1024x768, 1280x1024, 256 colores, 16 colores...
  • Tamaño de letra: small, medium, large...
  • Velocidad de la conexión: ADSL, módem...

Top 10 Mistakes in Web Design

  1. Bad search
  2. PDF files for online reading
  3. Not changing the color of visited links
  4. Non-scannable text
  5. Fixed font size
  6. Page titles with low search engine visibility
  7. Anything that looks like an advertisement
  8. Violating design conventions
  9. Opening new browser windows
  10. Not answering users' questions

Herramientas para pruebas de sitios web

  • Una labor muy intensiva en trabajo
  • Hay herramientas que comprueban sitios web para muchas cosas
    • Compatibilidad con navegadores
    • Problemas de rendimiento
    • Hipervínculos rotos
    • Conforme a estándares HTML
    • Corrección ortográfica del texto
