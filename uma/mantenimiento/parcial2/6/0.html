Documentación de las pruebas y su seguimiento

Objetivos

  • Propósito e importancia de la documentación
  • Documentos clave y relaciones entre ellos
  • Estructura y contenido clave de estos documentos
  • Generación automática y gestión de estos documentos

Para qué producir documentación ?

  • Para monitorizar y estimar el proceso
    • Para uso interno
      • Visibilidad del proceso
    • Para uso externo
      • Clientes (nivel de confianza)
      • Autoridades (certificaciones, auditorías)
  • Mejora del proceso
    • Mantenemos un cuerpo de conocimiento
    • Resumimos y mostramos datos para mejorar el proceso
  • Reutilización de los conjuntos de pruebas
    • Dentro del proyecto y entre proyectos

Tipos de documentos

  • Documentos de planificación
    • Descripción de la organización del proceso de calidad
    • Estrategias de organización y planes de proyecto
  • Documentos de especificación
    • Descripción de suites de pruebas y casos de prueba
      • Y otros artefactos para otras tareas relacionadas con la calidad
    • Especificaciones de diseño de pruebas
    • Especificaciones de casos de prueba
    • Checklists
    • Especificaciones de procedimientos de análisis
  • Informes
    • Detalles y resumen del análisis y resultados de las pruebas

Metadatos

  • Los documentos deben incluir metadatos para facilitar su gestión
    • Aprobación: personas responsables del documento
    • Historia del documento
    • Índice de contenidos
    • Resumen: relevancia y posibles usos del documento
    • Objetivos: propósito del documento (a quién va dirigido y por qué)
    • Documentos relacionados
      • Referencias a documentos y artefactos necesarios para entenderlo
    • Glosario: términos técnicos usados en el documento

Convención de nombres

  • Una buena convención de nombres ayuda a identificar documentos rápidamente
  • Estándar típico para nombres de documentos incluye palabras indicando:
    • Ámbito general del documento (proyecot y parte)
    • Tipo de documento (plan de pruebas, análisis de riesgos...)
    • Identidad del documento específico
    • Versión

Estrategia de análisis y pruebas

  • El documento de estrategia describe las guías de calidad para grupos de proyectos
    • Típicamente para una empresa u organización entera
  • Depende de la organización
  • Elementos clave: requisitos de calidad comunes
  • Puede depender de las condiciones de negocio
    • Un productor de software crítico puede necesitar satisfacer un conjunto mínimo de requisitos de fiabilidad definidos por una autoridad certificadora
    • Un departamente de software empotrado puede necesitar garantizar la portabilidad en líneas de productos
  • Establece requisitos para otros documentos de calidad

Plan de análisis y pruebas

  • Plan global y planes individuales
  • Estructura estandarizada
    • Elementos de análisis y pruebas: elementos a ser analizados o probados
    • Características a probar: características consideradas en el plan
    • Características a no probar: características no consideradas en el plan
    • Enfoque: enfoque global para el análisis y las pruebas
    • Criterio de éxito/fallo: reglas que determinan el estado de un artefacto
    • Criterio de suspensión y recuperación: condiciones para proceder a la suspensión de las actividades de prueba y análisis
    • Riesgos y contingencias: riesgos contemplados y planes de contingencia
    • Tareas y planificación: descripción de las tareas de análisis y pruebas
    • Personal y responsabilidades
    • Necesidades de entorno: software y hardware

Documentos de especificación de diseño de pruebas

  • Mismo propósito que otros documentos de diseño de software
    • Guía el desarrollo posterior
    • Prepara para el mantenimiento
  • Documentos de especificación del diseño de pruebas
    • Describen un conjunto de pruebas completo
    • Puede estar dividido
      • Organizadas por granularidad: pruebas de unidad, integración, sistema, aceptación...
      • Organizadas por objetivos: pruebas funcionales, estructurales, de rendimiento...
      • ...
    • Incluye información necesaria para
      • selección inicial de casos de prueba
      • mantenimiento de los casos de prueba
    • Identifica características a verificar
      • Referencias a documentos de análisis o diseño
    • Incluye descripciones de procedimientos de prueba y criterios de éxito/fallo
      • Referencias a andamiajes y oráculos
    • Incluye una lista de casos de prueba

Documento de especificación de casos de prueba

  • Diseño de prueba completo para cada caso de prueba individual
  • Define:
    • entradas de la prueba
    • condiciones de entorno necesarias
    • procedimiento para la ejecución de las pruebas
    • salidas esperadas
  • Indica:
    • elemento a ser probado (referencia a documento de diseño)
  • Describe dependencias con la ejecución de otros casos de prueba
  • Etiquetado con un identificador único

Informe de pruebas y análisis

  • Informa de los resultados de las pruebas y el análisis
  • Sirve para:
    • desarrolladores
      • identifica errores pendientes
      • planifica arreglos y revisiones
    • diseñadores de pruebas
      • estima y refina su enfoque
  • Lista con prioridades de los errores pendientes
    • núcleo del tratamiento de errores y procedimientos de reparación
  • Los informes de errores deben ser:
    • consolidados y categorizados para gestionar sistemáticamente el esfuerzo de reparación
    • priorizado para asignar esfuerzoa apropiadamente y manejar todos los errores

Informes resumen y logs detallados

  • Los informes resumen controlan el progreso y el estado
    • confirmación de que el ciclo construye-y-prueba funciona correctamente
    • proporciona información sobre dónde debemos poner atención
  • Incluye tablas resumen
    • conjuntos de pruebas ejecutados
    • número de fallos
    • clasifica los fallos en:
      • repetidos de ejecuciones de prueba anteriores
      • fallos nuevos
      • fallos que antes fallaban y ahora se ejecutan con éxito
  • Puede ser prescrito por una autoridad certificadora

Demasiado trabajo ?

    • Ten claro siempre el objetivo de un documento
    • No dediquen esfuerzo a documentos que no sean útiles
    • Parte de estos documentos puede ser automatizada
  • Documentación efectiva y de calidad
