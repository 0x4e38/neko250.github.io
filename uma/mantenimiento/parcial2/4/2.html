Mockito: un framework para mocking

Mockito

  • Framework open-source para Java
  • Desacopla el código de pruebas del sistema probado eliminando el patrón espera-ejecuta-verifica
  • Para simplificar las pruebas de unidad (integración) y proporcionar control sobre el contexto de ejecución permite reemplazar objetos reales con los que colabora el objeto probado con sustitutos falsos ("dobles")
  • Los dobles tienen el aspecto de los originales, pero no tienen dependencias con otros objetos

 Dobles

  • Tipos de dobles:
    • Dummy: objeto vacío pasado en una invocación
      • para satisfacer al compilador porque un método requiere un argumento
    • Fake: un objeto con una implementación funcional, pero simplificada, sólo para permitir la prueba (ej: base de datos en memoria)
    • Stub: objeto con un comportamiento "enlatado" para una prueba concreta (o un conjunto de pruebas)
    • Mock: objeto con la habilidad de:
      • tener un comportamiento esperado programado
      • verificar las interacciones en las que interviene
    • Spy: un mock creado como un proxy de un objeto real (unos métodos son simulados y otros redirigidos al objeto real)
  • Mockito permite crear mocks y spies

Cómo crear un mock

  • Creamos un mock con el método estático mock():
Flower flowerMock = Mockito.mock(Flower.class);

Métodos simulados

  • Habilidad de devolver valores determinados cuando un método es invocado
  • Usamos el par Mockito.when() - Mockito.thenReturn()
package flower;

import static org.mockito.Mockito.*;
import static org.junit.Assert.*;

import org.junit.Test;

public class SimpleStubbingTest {
    public static final int TEST_NUMBER_OF_LEAFS = 5;
    
    @Test
    public void shouldReturnGivenValue() {
        Flower flowerMock = mock(Flower.class);
        when(flowerMock.getNumberOfLeafs()).thenReturn(TEST_NUMBER_OF_LEAFS);
        
        int numberOfLeafs = flowerMock.getNumberOfLeafs();
        assertEquals(numberOfLeafs, TEST_NUMBER_OF_LEAFS);
    }
}

Funciones para especificar comportamiento

Método Descripción
thenReturn(T valueToBeReturned) Devuelve el valor indicado
thenThrow(Throwable toBeThrown) Lanza la excepción indicada
thenThrow(Class<? extends Throwable> toBeThrown) Lanza la excepción indicada
then(Answer answer) Utiliza código del usuario para responder
thenAnswer(Answer answer) Utiliza código del usuario para responder
thenCallRealMethod() Invoca un método real (para mock/spy parciales)
  • Métodos no void devuelven el valor por defecto para su tipo (null, 0, false, empty collection...)

El patrón prepara-actúa-comprueba

  • Las pruebas se dividen en tres partes
    • prepara (arrange, given)
      • inicialización y configuración del software a probar y de los mocks
    • actúa (act, when)
      • operación sujeta a pruebas (recomendado usar una sola operación)
    • comprueba (assert, then)
      • fase de comprobación y verificación

BDDMockito

import static org.mockito.BDDMockito.*;
import static org.mockito.Mockito.*;
import static org.junit.Assert.*;

import org.junit.Test;

public class Test {
    public static final int TEST_NUMBER_OF_LEAFS = 5;
    
    @Test
    public void shouldReturnGivenValueUsingBDDSemantics() {
        // given
        Flower flowerMock = mock(Flower.class);
        given(flowerMock.getNumberOfLeafs()).willReturn(TEST_NUMBER_OF_LEAFS);
        
        // when
        in numberOfLeafs = flowerMock.getNumberOfLeafs();
        
        // then
        assertEquals(numberOfLeafs, TEST_NUMBER_OF_LEAFS);
    }
}

Matching de argumentos

  • Por defecto, Mockito compara argumentos utilizando equals()
@Test
public void shouldMatchSimpleArgument() {
    WateringScheduler schedulerMock = mock(WateringScheduler.class);
    when(schedulerMock.getNumberOfPlantsScheduledOnDate(WANTED_DATE)).thenReturn(VALUE_FOR_WANTED_ARGUMENT);
    
    assertEquals(schedulerMock.getNumberOfPlantsScheduledOnDate(WANTED_DATE), VALUE_FOR_WANTED_ARGUMENT);
    
    assertEquals(schedulerMock.getNumberOfPlantsScheduledOnDate(ANY_OTHER_DATE), 0); // default value for int

@Test
public void test() {
    when(schedulerMock.getNumberOfPlantsScheduledOnDate(argThat(haveHourFieldEqualTo(7)))).thenReturn(VALUE_FOR_7);
    assertEquals(schedulerMock.getNumberOfPlantsScheduledOnDate(DATE_WITH_HOUR_7), VALUE_FOR_7);
}

private ArgumentMatcher<Calendar> haveHourFieldEqualTo(final int hour) {
    return new ArgumentMatcher<Calendar>() {
        @Override
        public boolean matches(Object argument) {
            return ((Calendar) argument).get(Calendar.HOUR) == hour;
        }
    };
}

Métodos void

@Test(expected = WaterException.class)
public void shouldStubVoidMethod() {
    WaterSource waterSourceMock = mock(WaterSource.class);
    doThrow(WaterException.class).when(waterSourceMock).doSelfCheck();
    waterSourceMock.doSelfCheck();
}
Método Descripción
doThrow(Throwable toBeThrown) Lanza la excepción especificada
doThrow(Class<? extends Throwable> toBeThrown) Lanza la excepción especificada
doAnswer(Answer answer) Utiliza el código del usuario
doCallRealMethod() Con spy
doNothing() No hace nada
doReturn(Object toBeReturned) Devuelve el valor especificado

Verificación del comportamiento

  • Una vez creado, un mock recuerda todas las operaciones realizadas sobre él
  • Podemos verificar estas operaciones con Mockito.verify(T mock)
    • Mockito.verify() comprueba que se invocó una sola vez
@Test
public void verifyVoidMethod() {
    WaterSource waterSourceMock = mock(WaterSource.class);
    waterSourceMock.doSelfCheck();
    verify(waterSourceMock).doSelfCheck();
}

Modos de verificación

Nombre Método de verificación
times(int wantedNumberOfInvocations) exactamente n veces
never() nunca
atLeastOnce() al menos una vez
atLeast(int minNumberOfInvocations) al menos minNumberOfInvocations veces
atMost(int maxNumberOfInvocations) a lo sumo maxNumberOfInvocations veces
only() el único método invocado
timeout(int millis) interaccionado en el rango de tiempo dado 
verify(waterSourceMock, never()).doSelfCheck();
verify(waterSourceMock, times(2)).getWaterPressure();
verify(waterSourceMock, atLeast(1)).getWaterTemperature();

Verificación del orden de invocación

@Test
public void shouldVerifyInOrderThroughDifferentMocks() {
    WaterSource waterSource1 = mock(WaterSource.class);
    WaterSource waterSource2 = mock(WaterSource.class);
    
    waterSource1.doSelfCheck();
    waterSource2.getWaterPressure();
    waterSource1.getWaterTemperature();
    
    InOrder inOrder = inOrder(waterSource1, waterSource2);
    inOrder.verify(waterSource1).doSelfCheck();
    inOrder.verify(waterSource2).getWaterPressure();
    inOrder.verify(waterSource1).getWaterTemperature();
}
