Objetivos

  • Objetivos de las pruebas de integración
    • Distinguir errores de integración de errores que deberían haber sido eliminados con pruebas de unidad
    • Entender la naturaleza de los errores de integración, y cómo prevenirlos y detectarlos
  • Estrategias de construcción y pruebas
    • Enfoques de ensamblaje y prueba incremental reduciendo esfuerzo y controlando riesgo

Pruebas de integración y pruebas de unidad

  • Purebas de unidad son un fundamento necesario
    • Con las unidades tenemos control y visibilidad máxima
    • Las pruebas de integración no pueden compensar malas pruebas de unidad
  • Las pruebas de integración pueden servir como una comprobación del proceso
    • Si se detectan errores de unidad durante las pruebas de integración, es que se han hecho mal las pruebas de unidad
    • Si ocurren errores de integración en interfaces entre módulos implementados correctamente entonces podemos identificar los problemas de descomposición modular y de especificación de interfaces

Errores de integración

  • Interpretación inconsistente de parámetros o valores
    • Ej: mezcla de unidades (metros/yardas)... Mars climate orbiter
  • Violaciones de rango de valores, capacidad o límites de tamaño
    • Ej: buffer overflow... librería estándar para lectura de PNGs, Apache 2 Web Server 2.0.35-2.0.50
  • Efectos laterales sobre parámetros o recursos
    • Ej: conflicto en fichero temporal (mal especificado), memory leak
  • Funcionalidad omitida o mal entendida
    • Ej: contabilización de web hits
  • Propiedades no funcionales
    • Ej: rendimiento imprevisto
  • Problemas con enlazado dinámico
    • Ej: llamadas incompatibles a métodos polimórficos

Estrategias para las pruebas de integración

  • Puede que alguna vez hayas oído eso de
    • Sí, he implementado el módulo A, pero no lo he probado aún, se probará con el módulo B cuando esté listo
  • Obviamente...
    • No se pensó en la estrategia para desarrollar las pruebas
    • El módulo A no se diseñó para ser probado
    • No se pensó cuál era el mejor orden para desarrollar y probar los módulos A y B

Planes de integración y pruebas

  • El plan de pruebas de integración guía y es guiado por el "plan de desarrollo" (parte de la arquitectura del sistema y del plan de proyecto)

Estrategias

Probamos tras integrar todos los módulos

  • Ventajas
    • No requiere ningún tipo de andamiaje
  • Inconvenientes
    • Observabilidad mínima
    • Capacidad de diagnóstico nula
    • Eficacia cero
    • Retroalimentación inadecuada
    • Alto coste de reparación
      • Cuanto mayor es el tiempo entre que se introduce el error y que este se repara mayor es el coste

Estrategias incrementales

  • Orientadas a la estructura
    • Los módulos se desarrollan, integran y prueban según la estructura jerárquica del proyecto
      • de arriba a abajo (top-down)
      • de abajo a arriba (bottom-up)
      • bocadillo (sandwich)
  • Orientadas a la funcionalidad
    • Los módulos se integran según las características de la aplicación
      • hebras (threads)
      • módulos críticos

De arriba a abajo (según relación de uso o inclusión)

  • Al top normalmente tendremos GUI, CLI, web app, etc.
  • No se necesitan drivers (o casi)
  • Requiere stubs (esbozo, boceto, ...)

De arriba a abajo progresivamente

  • Vamos añadiendo stubs de módulos invocados o usados en cada paso de la construcción según se necesiten

  • Conforme los módulos van reemplazando stubs con más funcionalidad es posible probar

  • Cuando la aplicación ha sido completada podemos probar toda la funcionalidad

De abajo a arriba

  • Comenzando desde las hojas de jerarquía de uso/invocación no necesitamos stubs (salvo para evitar circularidades)

  • Debemos construir drivers para cada módulo

  • Un módulo intermedio reemplaza a un driver y necesita su propio driver

  • Podemos tener varios subsistemas funcionando a la vez

  • Finalmente llegaremos a integrar el sistema completo

Bocadillo

  • Procediendo desde los extremos posiblemente necesitaremos menos drivers y stubs

  • Flexible y adaptable
  • Compleja de planificar
    • Normalmente sigue plan de desarrollo

Hebra

  • Una "hebra" es una poción de varios módulos que juntos proporcionan una funcionalidad del programa visible para el usuario

  • Integrando una hebra tras otra maximizamos la visibilidad por el usuario

  • Minimiza el número de stubs y drivers
  • El plan de integración puede ser complejo

Módulos críticos

  • Estrategia: empieza con el módulo de mayor riesgo
    • Necesario tener una estimación de riesgos
    • Pueden ser riesgos técnicos, riesgos de proceso, etc.
  • Proceso orientado a riesgos
    • Las pruebas de integración son una actividad para reducir los riesgos
    • Damos las malas noticias tan pronto como sea posible

Selección de estrategia

  • Las estrategias basadas en funcionalidades requieren mayor planificación
    • Las estrategias estructurales son más sencillas
    • Las pruebas basadas en hebras y módulos críticos proporcionan mayor visibilidad del proceso, especialmente en sistemas complejos
  • Posible combinar... habitual combinar en la práctica
    • Arriba-abajo, abajo-arriba o bocadillo son razonables para componentes y subsistemas relativamente pequeños
    • Combinaciones de hebra y módulos críticos son preferibles para grandes subsistemas

Andamiaje (Scaffolding)

  • Código que se produce para dar soporte a las actividades de desarrollo
    • No es parte del "producto" que ve el usuario
  • Incluye
    • test harnesses (arnés)
    • drivers (clases de prueba)
    • stubs (esbozo, boceto...)

Andamiaje

  • Driver de pruebas
    • Programa principal para ejecutar pruebas
      • Puede ser creado antes de existir el programa principal "real"
      • Proporciona más control que el programa principal "real"
  • Stubs de pruebas
    • Substituto para llamadas a funciones/métodos/objetos
  • Arnés de pruebas (harness)
    • Substituto para otras partes del entorno implantado
      • Ej: simulación software de un dispositivo hardware
    • Podemos encontrarnos usos de arneses para referirse tanto a drivers, stubs... como a substitutos de otros componentes
